# 个人作品集搭建

> 用 GitHub 打造技术个人品牌——Profile README、Pages 作品集、Pinned Repository 的完整搭建方案。

## 概述

场景：你是一名前端工程师，准备跳槽到更好的团队。HR 在筛选简历时通常会搜索候选人的 GitHub 主页。一个精心打理的 GitHub Profile 可以在几秒内传递"这个人对代码是认真的"这一信号，比简历上的自我评价有力得多。

GitHub 提供了多种展示个人技术能力的方式：Profile README 让你的主页变成一份动态简历，GitHub Pages 让你搭建零成本的个人网站，Pinned Repository 让访客第一眼就看到你最得意的项目。三者配合使用，可以构建一个完整的个人技术品牌。

> [!NOTE]
> GitHub Profile 页面（`github.com/<username>`）默认只显示你的贡献热力图、仓库列表和基本信息。通过创建与用户名同名的仓库并添加 README.md，你可以在 Profile 页面顶部展示自定义内容。这个功能于 2020 年推出，已成为技术从业者个人品牌建设的事实标准。

本指南将带你从零搭建一套完整的 GitHub 个人作品集，涵盖 Profile README 编写、Pages 站点部署和项目展示优化。每个部分都可以独立完成，也可以组合使用以达到最佳效果。

## 核心操作

### 搭建 Profile README

Profile README 是投入产出比最高的个人品牌建设手段——一个 Markdown 文件就能彻底改变你的 GitHub 主页。

**1. 创建同名仓库**

```bash
# 创建与 GitHub 用户名完全一致的仓库
# 假设你的用户名是 octocat
gh repo create octocat --public --description "Profile README"
```

或者通过浏览器：访问 https://github.com/new，仓库名输入你的 GitHub 用户名，选择 Public，点击 Create repository。

**2. 编写 README.md**

在仓库根目录创建 `README.md`，以下是一个推荐的 Profile README 结构：

```markdown
# 你好，我是 [你的名字] 👋

> 全栈工程师 | 开源爱好者 | 技术博客作者

- 🔭 目前在 [公司/项目] 从事 [工作内容]
- 🌱 正在学习 [技术方向]
- 💬 欢迎和我讨论 [擅长领域]
- 📫 联系方式：[邮箱/社交媒体]
```

### 使用自动化工具增强 Profile

静态的 Profile README 虽然好用，但需要手动更新。借助 GitHub Actions，可以让 Profile 自动展示最新动态。

**自动更新最近活动：**

```yaml
# .github/workflows/profile-update.yml
name: Update Profile README
on:
  schedule:
    - cron: "0 0 * * *"  # 每天 UTC 零点执行
  workflow_dispatch:       # 也支持手动触发

jobs:
  update-readme:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: 更新最近活动
        uses: jamesgeorge007/github-activity-readme@master
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        with:
          MAX_LINES: 5
          COMMIT_NAME: "github-actions[bot]"
          COMMIT_EMAIL: "github-actions[bot]@users.noreply.github.com"
```

此 Action 会自动在 `<!--START_SECTION:activity-->` 和 `<!--END_SECTION:activity-->` 之间填充你最近的 GitHub 活动，如 PR 合并、Issue 评论、Star 操作等。

**常用 Profile 增强工具：**

| 工具 | 功能 | 仓库 |
|------|------|------|
| github-readme-stats | 展示 GitHub 统计卡片 | `anuraghazra/github-readme-stats` |
| github-readme-streak-stats | 展示连续贡献天数 | `DenverCoder1/github-readme-streak-stats` |
| github-readme-activity-graph | 展示贡献活动图 | `ashutosh00710/github-readme-activity-graph` |
| github-profile-trophy | 展示成就奖杯 | `ryo-ma/github-profile-trophy` |

> [!TIP]
> 不要在 Profile 中堆砌太多统计卡片和动画效果。招聘者浏览一个 Profile 通常只花 10-30 秒，信息密度和可读性比花哨更重要。推荐最多使用 2-3 个增强工具。

### 搭建 GitHub Pages 作品集网站

Profile README 适合快速展示，而 GitHub Pages 则能承载更完整的作品集——项目展示、技术文章、工作经历等。参考 [GitHub Pages 建站](../05-文档与知识管理/02-GitHub-Pages建站.md) 中的基础操作，这里重点讲解作品集场景的具体实践。

**1. 选择主题或框架**

| 框架 | 特点 | 适合场景 |
|------|------|---------|
| Jekyll + al-folio | 学术风格，内置论文列表 | 研究者、博士生 |
| Hugo Academic | 功能丰富，主题众多 | 学术和技术作品集 |
| Next.js | 高度可定制 | 前端工程师展示技术能力 |
| Astro | 性能优异，支持多框架 | 博客 + 作品集混合场景 |
| 纯 HTML/CSS | 最简单，完全掌控 | 极简风格作品集 |

**2. 使用 Jekyll 搭建作品集**

```bash
# 创建用户站点仓库
gh repo create <username>.github.io --public

# 克隆并初始化 Jekyll
git clone https://github.com/<username>/<username>.github.io.git
cd <username>.github.io

# 创建 Gemfile
cat > Gemfile << 'EOF'
source "https://rubygems.org"
gem "github-pages", group: :jekyll_plugins
EOF

# 创建基础配置
cat > _config.yml << 'EOF'
title: 你的名字
description: 全栈工程师作品集
theme: minima
markdown: kramdown
plugins:
  - jekyll-seo-tag
EOF

# 创建首页
cat > index.md << 'EOF'
---
layout: home
title: 首页
---

# 关于我

这里是一段自我介绍...

# 项目作品

## 项目名称 A
技术栈：React + TypeScript + Node.js
简短描述...

## 项目名称 B
技术栈：Vue.js + Python
简短描述...
EOF

# 提交部署
git add .
git commit -m "feat: 初始化作品集站点"
git push origin main
```

**3. 配置 GitHub Actions 自动部署**

即使使用 Jekyll，也推荐通过 GitHub Actions 部署，可以获得更好的构建控制：

```yaml
# .github/workflows/deploy.yml
name: Deploy Pages
on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/cache@v4
        with:
          path: vendor/bundle
          key: gems-${{ hashFiles('**/Gemfile.lock') }}
      - run: bundle install
      - run: bundle exec jekyll build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: _site

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
    steps:
      - uses: actions/deploy-pages@v4
```

### 精选展示项目（Pinned Repository）

Profile README 和 Pages 作品集都需要手动维护内容，而 Pinned Repository 则是零维护的展示方式。访客打开你的 GitHub 主页时，Pinned 仓库会出现在最显眼的位置。

**固定项目的操作步骤：**

1. 访问你的 GitHub Profile 页面（`github.com/<username>`）。
2. 在 **Pinned** 区域点击 **Customize your pins**。
3. 最多选择 6 个仓库进行固定。
4. 点击 **Save pins**。

**选择 Pinned 项目的策略：**

> [!NOTE]
> Pinned 项目的选择应该服务于你的职业目标，而非简单展示 Star 数最多的项目。如果你希望求职前端岗位，就固定前端相关的项目；如果你希望以开源贡献者身份获得认可，就固定你最活跃的开源贡献仓库。

推荐的 6 个 Pin 位分配：

- **3 个最佳原创项目** —— 代码质量高、文档完善、有实际用户的项目。
- **1-2 个开源贡献** —— 展示你参与社区协作的能力。
- **1 个学习/实验项目** —— 展示你持续学习的态度，可以是新技术的探索性项目。

**优化项目展示效果：**

固定后，项目的 README 封面就是展示效果的关键。确保每个 Pinned 项目：

1. 有清晰的标题和一句话描述。
2. 有状态徽章（构建状态、版本号）。
3. 有安装和使用的快速开始指南。
4. 有预览截图或 Demo 链接。

## 进阶技巧

### 使用 GitHub README 的 "关于" 侧边栏

Profile README 仓库的 About 区域（右侧边栏）可以利用起来：

1. 进入 Profile 仓库的首页。
2. 点击右上角的齿轮图标（Settings 齿轮）。
3. 在 **Website** 中填入你的作品集网站地址。
4. 在 **Topics** 中添加技术标签，如 `frontend`、`react`、`typescript`。
5. 勾选 **Releases** 和 **Packages**（如果有）。

### 利用 GitHub成就（Achievements）

GitHub 会自动在你的 Profile 上展示获得的成就徽章，如：

- **Arctic Code Vault Contributor** —— 向 2020 年 Arctic Code Vault 中存档的仓库贡献过代码。
- **Pull Shark** —— 成功合并了一定数量的 Pull Request。
- **YOLO** —— 直接 push 到生产分支（不推荐，但确实是个成就）。
- **Galaxy Brain** —— 获得了有价值的讨论回答标记。

保持活跃贡献，这些成就会自然积累，为你的 Profile 增添社交证明。

### 在简历和 LinkedIn 中引用 GitHub

Profile README 和作品集搭建完成后，不要忘了在其他渠道引流：

- 简历中添加 GitHub 链接（放在联系方式区域）。
- LinkedIn 个人资料中添加 GitHub URL。
- 技术博客文章中引用你的开源项目。
- 面试时展示项目代码和贡献历史。

### 使用自定义域名

如果你有自己的域名，可以将其绑定到 GitHub Pages 作品集：

1. 在 Pages 仓库的 **Settings > Pages > Custom domain** 中填写你的域名。
2. 在域名注册商处添加 CNAME 记录指向 `<username>.github.io`。
3. 勾选 **Enforce HTTPS**。

```bash
# 在仓库中创建 CNAME 文件
echo "www.yourdomain.com" > CNAME
git add CNAME
git commit -m "chore: 配置自定义域名"
git push
```

> [!WARNING]
> 绑定自定义域名后，DNS 传播可能需要数小时甚至 24 小时才能生效。在此期间网站可能暂时无法访问。务必勾选 **Enforce HTTPS**，否则访客通过 HTTP 访问时不会自动跳转到安全连接。如果使用 Apex 域名（如 `example.com` 而非 `www.example.com`），需要配置 A 记录而非 CNAME 记录，具体 IP 地址参考 GitHub Pages 官方文档。

## 常见问题

### Q: Profile README 没有显示在主页上？

确认仓库名与你的 GitHub 用户名完全一致（包括大小写），且仓库设为 Public。README.md 必须在仓库根目录或 `docs/` 目录中。如果仓库名包含空格或特殊字符，需要使用 URL 编码后的名称。

### Q: GitHub Pages 站点构建失败怎么办？

进入仓库的 **Actions** 标签页查看构建日志，常见原因包括：Gemfile 依赖版本不兼容、Markdown 语法错误、配置文件格式错误。使用 GitHub Actions 部署时，检查 workflow 文件的 `permissions` 是否正确配置了 `pages: write` 和 `id-token: write`。

### Q: 如何让 Profile README 中的统计数据实时更新？

使用 Vercel 部署 `github-readme-stats` 的个人实例，而不是使用公共 API 端点。公共端点有缓存延迟和请求限制。将统计数据图片的 URL 中的 `?cache_seconds=` 参数设置为需要的刷新频率（最低 1800 秒）。

### Q: 应该固定哪些仓库？

根据你的目标受众来选择。如果目标是求职，固定与你目标岗位相关的项目；如果目标是开源影响力，固定你最活跃的开源项目和获得最多 Star 的贡献。Pinned 项目应该展示你的技术广度和深度。

### Q: 作品集网站需要包含哪些内容？

至少包含：自我介绍（1-2 段）、技能清单、项目展示（3-6 个，每个附链接和简要说明）、联系方式。可选内容：工作经历、教育背景、技术博客文章列表、演讲或分享记录。保持内容精简，避免信息过载。

### Q: 如何让个人品牌保持一致？

确保 GitHub 用户名、Profile 头像、Bio 描述与 LinkedIn、Twitter/X 等平台保持一致。使用相同的头像和简介可以让招聘者快速确认你的身份。在 Profile README 中添加其他平台的链接，形成完整的个人品牌网络。

### Q: 零贡献历史怎么办？

如果 GitHub 贡献热力图看起来比较空旷，可以从以下几个方面快速改善：将日常学习的代码练习推送到 GitHub（如 LeetCode 题解仓库）；参与开源项目的文档改进和 Bug 修复（这类贡献门槛低）；维护一个技术笔记仓库，将学习心得公开分享。

### Q: 需要花钱购买域名吗？

不是必须的。`<username>.github.io` 域名已经足够专业，且完全免费。如果你希望使用自定义域名，普通 `.me` 或 `.dev` 域名年费约 10-15 美元。对于学生，GitHub Student Developer Pack 提供免费域名。
